### [2026-04-19] 选定应用场景：游戏装备交易系统

- **背景**：需要从课程文档列出的场景中选择一个，作为本组的开发主题
- **选项**：
  - 学生选课系统
  - 图书借阅系统
  - 个人财务追踪
  - **游戏装备交易系统**
  - 健身追踪
  - …
- **决定**：游戏装备交易系统
- **理由**：
  - 题材有趣，组员有共同兴趣，开发动力更足
  - 天然包含丰富的实体关系（玩家 / 物品 / 挂单 / 交易），便于演示组合与多态
  - 物品分类天然对应**树结构**；价格区间查询天然对应 **BST**；玩家背包天然对应**链表**；ID 查询天然对应**哈希表**——课程要求的所有数据结构都有合理落点
- **影响**：所有模块；后续 `Item / Player / Listing / Transaction` 等核心实体的命名都基于此

### [2026-04-19] 持久化采用 JSON 文件而非 SQLite / 关系数据库

- **背景**：作业要求"file storage"，需要决定文件格式
- **选项**：
  - JSON（多文件，按实体分）
  - SQLite（单文件数据库）
  - pickle（Python 原生序列化）
  - 自定义文本格式
- **决定**：JSON，按实体拆成多个文件（`players.json` / `items.json` / `market.json` / `transactions.json` / `catalog.json`）
- **理由**：
  - 作业重点是**自实现数据结构**——若用 SQL，索引 / 排序 / 范围查询都被数据库接管，自实现的 BST / HashMap 就失去了用武之地
  - JSON 可读，调试时直接打开文件就能看；pickle 是二进制，难以排查问题
  - 数据规模小（百到千条），无并发需求，文件读写完全足够
  - 多文件结构与领域模型对应，加载逻辑清晰
- **影响**：`services/persistence.py`、所有模型类的 `to_dict / from_dict`、`data/` 目录结构

### [2026-04-19] 物品的 `stats` 字段使用动态 object 而非固定 schema

- **背景**：物品分为武器 / 工具 / 装备 / 消耗品 / 杂项五大类，各类属性差异很大，需要决定如何在 JSON 中存储
- **选项**：
  - 固定 schema：所有物品都存所有可能字段，不适用的填 `null`
  - 动态 object：`stats` 字段按类别存不同字段
  - 拆成多个 JSON 文件：每个类别一个文件
- **决定**：动态 object
- **理由**：
  - 固定 schema 字段大量为 null，可读性差且浪费空间
  - 多文件拆分增加加载复杂度，且物品的"统一身份"被割裂
  - 动态 object 天然契合 **多态反序列化**——`from_dict` 根据 `category` 实例化对应的 `Item` 子类，是 OOP 多态的优雅展示点
- **影响**：`Item` 抽象基类及其所有子类、`persistence.py` 的反序列化逻辑、`data/stats-design.md` 字段定义

---

### [2026-04-19] 物品分类采用点号路径字符串（如 `weapon.sword`）

- **背景**：物品分类是两级树（大类 / 子类），需要决定 `category` 字段格式
- **选项**：
  - 嵌套对象：`{"category": "weapon", "subcategory": "sword"}`
  - 点号字符串：`"weapon.sword"`
  - 数字 ID：`category_id: 11`
- **决定**：点号字符串
- **理由**：
  - 与 `catalog.json` 树的根→叶路径**一一对应**，便于互查
  - 单字段，比嵌套对象更紧凑
  - 字符串可读，比数字 ID 不需要查表
  - 杂项无子类时直接写 `misc`，扁平结构兼容良好
- **影响**：`Item.category` 属性、`catalog.json` 树遍历逻辑、按分类筛选挂单的服务方法

---

### [2026-04-19] 装备 / 武器引入耐久度 + 穿戴状态；消耗品 / 杂项支持堆叠

- **背景**：物品的真实游戏感与系统复杂度的权衡
- **选项**：
  - 全部物品都简单存储（无耐久 / 无穿戴 / 无堆叠）
  - 全套机制都引入
  - 部分引入
- **决定**：按物品分类差异化引入（全套）
  - 武器 / 工具 / 装备：耐久度
  - 武器 / 装备：穿戴状态
  - 消耗品 / 杂项：堆叠
- **理由**：
  - 真实感强，能演示更丰富的业务规则（耐久=0 不能挂单 / 穿戴中不能丢弃）
  - 业务规则交叉表（见 `stats-design.md`）正好是异常处理与多态行为的良好素材
  - 增加的实现复杂度可控，不会显著拖慢开发进度
- **影响**：所有 Item 子类、`Inventory.add/remove`、`Market.list/buy` 的校验逻辑、自定义异常类
